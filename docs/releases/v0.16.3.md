## Root-cause repair for the wrong HQ address

An upgraded client could preserve an encrypted credential for an older HQ
while Setup submitted a new enrollment request to the administrator-entered
server. The resident agent gave the old enrolled credential priority over the
new pending request.

For example:

- Old stored HQ: `https://192.168.1.9:8443`
- Intended HQ entered in Setup: `https://192.168.1.12:8443`
- Client computer: `192.168.1.13`

The request reached `.12`, but the background connector restarted against
`.9`. HQ `.12` therefore showed the request but received no telemetry after
approval, and protected changes called the maintenance API on the wrong
server.

The corrected transaction now:

1. Contacts and verifies the administrator-selected HQ.
2. Submits and securely stores the new pending enrollment request.
3. Removes the previous encrypted HQ credential only after the new server
   accepts that request.
4. Removes stale connector-state metadata for the previous address.
5. Starts the resident approval poller against the new HQ.
6. Stores the approved device token and begins telemetry on the selected
   server.

Regression coverage creates a preserved `.9` relationship, requests a new
local HQ, verifies the old credential and connector state are removed, approves
enrollment, sends telemetry, and executes an allowlisted command.

## Upgrade-safe state and configuration

Endpoint Setup preserves configuration, identity, encrypted credentials,
quarantine, logs, scan history, signatures, connector state, DNS state,
device-control state, monitoring metadata, threat intelligence, and update
state under an administrator-restricted backup.

HQ Setup preserves the configuration, TLS certificate, SQLite fleet database,
update repository, approvals, telemetry, maintenance records, and operational
history.

Versioned schema migrations deep-merge new defaults while retaining existing
values. An upgrade stops instead of silently replacing data when preservation
fails.

## HQ data and operations policy

The HQ dashboard now exposes discrete controls for:

- 30, 90, or 365-day data retention.
- Endpoint offline alerts after 1, 5, or 15 minutes.
- Administrator sessions lasting 1, 4, 8, 12, or 24 hours.
- 5, 10, or 20 failed login attempts per minute per address.
- Default maintenance passwords lasting 5, 10, 30, or 60 minutes.
- Default maintenance passwords allowing 1, 3, or 10 uses.
- Automatic deployment of newer trusted signed client updates.

Retention cleanup covers telemetry snapshots, completed command history,
reviewed enrollment requests, completed maintenance requests, and expired or
revoked maintenance records. Current device state and active security records
remain available.

## Endpoint compatibility and protected settings

Protected controls identify whether they:

- Work immediately in standalone mode.
- Require an HQ maintenance password.
- Are blocked while HQ is offline.
- Are checking legacy capability metadata on use.
- Explicitly require a newer compatible HQ.

Compatibility failures now name the actual HQ URL being contacted. Missing
capability metadata is treated as unknown instead of automatically unsupported,
which prevents false upgrade warnings between separate client and server PCs.

## FreshClam TLS reliability

SentryLoom exports the current-user and local-machine Windows trusted root
certificate stores into a private PEM bundle and supplies it to FreshClam
through `CURL_CA_BUNDLE`. Certificate verification remains enabled.

Diagnostics distinguish common causes such as incorrect system time,
enterprise TLS inspection, and missing trusted-root policy.
